# Aparatus Barber — Handoff técnico

## Estado atual

Este branch contém mudanças não commitadas. O marketplace B2C legado foi removido em favor do SaaS multi-tenant, mas a entrega ainda requer estabilização e validação completa.

Implementado nesta sessão:

- Multi-tenancy fail-closed com `AsyncLocalStorage`, Prisma scoped client e host-based tenant resolution.
- Schema/migrations para organizações, locais, equipe, serviços, clientes, agenda, regras, assinaturas e eventos Stripe.
- Motor de disponibilidade por profissional, horário, ausência, período fechado, buffers, lead time e timezone do tenant.
- Booking guest com hold de 30 minutos para pagamento online, Stripe Connect Express e confirmação idempotente por webhook.
- SaaS billing com Starter (€39), Growth (€79), Pro (€149), Stripe Tax, trial de 14 dias, grace period de 7 dias e Customer Portal.
- BullMQ/Redis + Resend para confirmação, cancelamento e lembrete 24h, executados por worker separado.
- Dockerfile e Docker Compose com web, worker, Postgres e Redis.
- Dashboard com shell RBAC, agenda semanal, clientes, serviços, equipe, billing e configurações de booking.

## Retomada

1. Verifique o worktree antes de editar:

   ```powershell
   git status --short
   ```

2. Restaure dependências e Prisma:

   ```powershell
   $env:CI='true'
   pnpm install
   pnpm prisma generate
   ```

3. Valide em banco limpo:

   ```powershell
   docker compose up -d postgres redis
   pnpm db:migrate:deploy
   pnpm typecheck
   pnpm test
   ```

4. Rode a stack local completa após configurar `.env`:

   ```powershell
   docker compose up --build
   ```

## Escala progressiva para 1.000+ usuários

- Web e worker devem permanecer stateless e escalar horizontalmente no Railway.
- Postgres é a fonte de verdade: índices de agenda, pooling, backups/PITR e restauração testada.
- Redis serve apenas BullMQ, cache de disponibilidade, rate limit e locks curtos; nunca é fonte de verdade de booking.
- Adicionar cache de disponibilidade por tenant/dia/serviço com TTL curto e invalidação em booking, ausência, horário, fechamento ou regra.
- Garantir idempotência por Stripe event ID, Checkout Session, job ID por booking/evento e operações mutáveis.
- Adicionar logs JSON com request ID, organization ID e booking ID; health checks, métricas de fila, alertas Stripe e monitoramento de latência.
- Aplicar rate limiting no wizard público e proteger criação de holds contra abuso.
- Manter módulos de domínio claros: `billing`, `booking`, `scheduling`, `notifications`, `tenant` e `dashboard`. Components não acessam Prisma diretamente; actions validam/autorizam e chamam serviços.
- Só introduzir novos bancos, Kafka, Kubernetes ou sharding após métricas demonstrarem necessidade.
